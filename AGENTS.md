# workflow

这个是只由人维护的项目工作流，包含工作流程和项目规范

## 项目维护

### 要使用的skill

1. docs-graph 用来维护项目概况 git提交时跟进 可以给subagent写
2. project_points 写入[可复用技术点、证据状态、假设、否定结果] 可以subagent写 主agent看
3. 论文skill 优先论文使用作为可靠信息来源 优先查相关领域期刊 最好多方验证

### 文件交互规范

1. 各处manual.md和本文件完全由人手动维护，agent不更改
2. 每次用户验证阶段性成果之后更新docs_graph状态,docs_graph作为项目级别的各模块状态概述
3. docs_graph/agent_notes.md 中维护项目进度概况和agent自动纠偏
4. 项目进度从docs_graph/agent_notes.md中了解，知识在project_points中维护和读取
5. 创建文件夹和文件的时候使用通用命名并处理好新文件/文件夹的层级 维护文件系统整洁

### 人机交流规范

1. 不必讨好鼓励用户，不用多余的表情包和废话，这里是项目场景优先完成项目
2. 如果用户说了“我们先讨论”“先讨论”“先不改动”这类表达，在用户明确要求更改之前不要改动文件，可以询问用户是否更改
3. 如果有影响较大的不确定的内容（涉及方向/不可逆/大量文件修改）优先停止然后询问用户，不擅自决定
4. 如果用户过度设计根据KISS（Keep It Simple & Stupid）原则质疑其是否必要，如果用户忽略了某种考量质疑其是否缺乏考量

### 典型工作流程

1. 用户完成本文件的编辑适配工作流
2. agent读取本文件并从docs_graph/agent_notes.md了解进度和用户习惯
3. 用户与agent交流实现，完成manual.md的写入（agent只看不写）
4. agent开始根据manual.md与用户直接指示完成manual.md中要求的内容
5. 使用无上下文subagent审查工作（给其项目基本信息和需求 要求严格审查）如果有明显问题则返回3/4 无或者缺失信息进入6
6. 汇报进度，用户介入补全信息或者评估产物
7. 根据用户整理信息对话特征更新docs_graph/agent_notes.md（遵从KISS），随后若本轮有用户认可的产物就阶段性提交，无则不提交然后进入3/4
